# 11 設計思想メモ: 種→苗→生産物の拡張

## 連なりの発想

種・苗・生産物(生鮮野菜)は栽培の時間軸に沿って自然につながる。
統合アプリ(aiseed)の品目ハブでは、品目(例: 三浦大根)ごとに
種・苗・大根そのもの・大根の料理(将来)が縦に並び、
「種を手に入れる→育てる→食べる」という動線になる。

出品機能自体は seed(個人の直売・余り野菜)と shop(店舗の産直)が持つ。
統合アプリは品目軸で束ねて見せるだけ。よって生産物対応の器は
seed・shop の両方に要る(item_kind に produce を追加済み)。

## 段階の切り方(重要な歯止め)

**今やること: 生鮮野菜まで。** 種・苗(種苗法の世界)+ 生鮮野菜。
生鮮野菜は郵送時に食品表示(名称・原産地・生産者・収穫日・保存方法)、
事業者の郵送は特商法表示、で対応できる。直接受け渡しなら表示は軽い。

**やらないこと(将来の別ドメイン):**
- 加工品(漬物・ジャム・乾燥野菜など)。食品表示基準のフルセット
  (原材料・添加物・アレルゲン・期限・製造者)+ 製造の営業許可/届出が
  絡む別世界。種苗法・生鮮の枠では扱えない
- 許可・届出を要する食品全般

Claude Code は produce として生鮮野菜までを実装し、加工品・許可要食品の
機能(成分表示・許可管理等)を先回りで作らないこと。実際に需要が出て、
食品表示法を種苗法と同じ深さで調べ直してから、別フェーズで設計する。

## 受け渡しと支払いの原則

- 受け渡し: 直接 / 郵送。直接受け渡しは表示が軽く、支払いリスクも小さい
- 支払い: **後払いを原則(既定)**。種は単価が安く、郵送で住所も分かるため、
  売り手が負う未払いリスクは小さく許容できる(野口のタネ・たねの森も後払い)。
  前払い・着払いも当事者協議で可。方法・金額・タイミングはアプリ外で決め、
  アプリは既定表示と規約の注意喚起のみ。決済機能は持たない
- 後払い原則のため、前払い前提の仕掛け(会員限定の前払い解禁・保証金・
  発送方法の縛り・第三種郵便の発送確認問題)は設けない。
  買い手が先に払わないので、それらの防御が不要になる

## 収益との関係

年会費(個人)・出店プラン(店舗)はコミュニティ運営の対価。
取引手数料は取らない(金額把握=決済に近づくため)。会費・プランは
アプリ外決済(振込・請求書)で、アプリは会員/有効フラグを見るだけ。
